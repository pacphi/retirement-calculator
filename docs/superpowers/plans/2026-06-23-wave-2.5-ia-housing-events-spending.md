# Wave 2.5 — IA, Housing↔Places Wiring, Event Taxonomy & Spending-Basis Reframe

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development
> (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use
> checkbox (`- [ ]`) syntax for tracking.

**Goal:** A front-end/UX cleanup pass that lands BEFORE Wave 3: reorder the steps into a clean 1–10
sequence, split the work-vs-retire location controls into the steps where users expect them, wire the
household's actual housing (mortgage / owned / rent + relocation sell/keep) into the **Places**
comparison (today a confirmed dead wire), fix the family-milestone event taxonomy (all 7 defaults
silently fall back to "gift"), and reframe the confusing "28% of income" spending control into a
familiar total-replacement ratio — all behavior-neutral on the default headline except where a
re-baseline is explicitly called out.

**Architecture:** Five mostly-independent fixes. (1) **IA reorder** is pure JSX relocation in the
composition root — no engine touch. (2) **Location split** moves the work-state picker into Step 1
(Household) and lifts retire-location / relocation-year / retirement-state-tax / rate-override into a
new grouped **Retirement place & taxes** step between Pension and Inheritance; the underlying state
keys (`workLoc`, `retireLoc`, `relocationYear`, `stateCode`, `stateRate`) are unchanged, so the
engine is untouched. (3) **Housing→Places** extracts the retirement-dwelling resolver that already
lives inside `simulate.js` into a shared `housing.js` function, then feeds the resolved household
housing cost into `usePlan.js`'s `locRows` so the comparison reflects your real situation. (4) **Event
taxonomy** assigns explicit `type`s to the defaults and reconsiders the category set. (5)
**Spending-basis reframe** keeps the engine's stored value as the non-housing share (zero engine math
change → headline-neutral) and makes the slider a presentation transform that reads/writes a
total-replacement %.

**Tech Stack:** React 19, Vite, Vitest 4 + @testing-library/react 16, Recharts 3, pure ES-module
finance engine under `src/finance/**` re-exported by `src/calculatorCore.js`; theme tokens in
`src/components/theme.js` (brass `#B5852C` / viridian `#1E7A5E` / clay `#BE4A2B`;
Inter / Newsreader / JetBrains Mono).

## Decisions taken (confirmed with the user)

1. **Sequence — Wave 2.5 BEFORE Wave 3**, on its own branch (`feat/wave-2.5-ia-housing-events`).
   Wave 3's plan (`docs/superpowers/plans/2026-06-23-wave-3-engine-depth.md`) slots its new steps
   (Contributions near Step 1, Return-model + Guardrails under Advanced/Spending) into this settled IA.
2. **Step order — adopt the proposed clean 1–10** (see File structure). Work-state → Step 1; Housing
   → Step 2 (right after Household); a new **Retirement place & taxes** step at position 5 (between
   Pension and Inheritance).
3. **Housing→Places — reflect the resolved dwelling's TRUE cost composition.** Places must consume the
   same retirement-dwelling resolution the projection uses, and the substituted cost depends on the
   resolved tenure (not a simple own-vs-rent binary):
   - **Own outright in the retirement location** → NO monthly rent; carrying cost = property tax +
     insurance + maintenance only.
   - **Mortgage carried into retirement** (a balance remained at the relocation year and the work home
     was NOT sold) → deflated P&I + carrying cost, until payoff.
   - **New mortgage taken in the retirement location** → model THAT mortgage's P&I + carrying cost —
     it differs from local rent.
   - **Rent in retirement** → the chosen rent (each candidate location's local basket rent stands).
   `housingCostForYear` already computes all of these; the resolver picks the dwelling. For the
   cross-location comparison, an owned/mortgaged home is a fixed cost you "bring with you" across
   candidate locations; a renting household sees each location's local rent. Surface the resolution
   tree explicitly in the Housing step (sell vs keep ask when a mortgage balance remains at relocation
   and the retirement location differs; sell → market value + taxable event; keep → carrying cost
   continues). Caption every simplification.
4. **Event taxonomy — fix types + rethink the set.** Assign explicit `type` to all 7 defaults
   (vehicle replacement & home upkeep → `purchase`, weddings/529 → `gift`, home-help → `gift`).
   Keep `type` as labeling unless a behavior should genuinely differ; reconsider the category list
   (gift / purchase / windfall, plus whether "recurring expense" deserves its own label).
5. **Spending basis — total-replacement slider, engine stores non-housing (headline-neutral).** The
   stored engine field stays the non-housing share (`targetPct`, default `0.28` — unchanged, so the
   projection is byte-identical). The UI control becomes a total-replacement % that reads/writes via
   `total = nonHousing + year-1 housing`; default displays ≈40%. A prominent breakdown shows total /
   housing / everything-else; mortgage payoff lowers the total over time; the smile provides decline.
